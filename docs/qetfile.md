# Qetfile Guide

A **Qetfile** is a TOML file that declares the desired software state of a system. You can use it to replicate an environment on a new machine, share your setup with colleagues, or keep your packages under version control.

---

## File Locations

| Location | Purpose |
|---|---|
| `~/.config/qet/Qetfile` | **Global Qetfile** — automatically synced by `qet add` / `qet remove` |
| `./Qetfile` | **Project Qetfile** — used by `qet sync` (in the current directory) |

> **Source:** [`qet/config.py` — `QETFILE_PATH`](../qet/config.py#L18) · [`get_qetfile()` / `save_qetfile()`](../qet/config.py#L184-L191)

---

## Format

A Qetfile contains an array of `[[packages]]` tables. Each entry has a `qet_name` and optionally a `method`.

```toml
# Qetfile

[[packages]]
qet_name = "@utils/htop"
method   = "apt"

[[packages]]
qet_name = "@microsoft/vscode"
method   = "flatpak"

[[packages]]
qet_name = "@python/uv"
method   = "script"
```

| Field | Required | Description |
|---|---|---|
| `qet_name` | Yes | The canonical qet name of the package. |
| `method` | Recommended | The installation method to use. If omitted, qet uses its priority list. |

## Automatic Sync

The global Qetfile at `~/.config/qet/Qetfile` is **automatically kept up to date** by qet:

- **`qet add`** appends the newly installed package (if not already listed).
- **`qet remove`** removes the entry for the uninstalled package.

You never need to manually edit the global Qetfile — it is always a reflection of what `qet` manages.

> **Source:** [`qet/commands.py` — inside `add_package()`](../qet/commands.py#L133-L141) and [`remove_package()`](../qet/commands.py#L199-L205)

## Exporting a Qetfile with `snapshot`

Generate a Qetfile from your currently installed packages:

```bash
# Print to stdout
qet snapshot

# Save to a file
qet snapshot > Qetfile

# Save to dotfiles
qet snapshot > ~/dotfiles/Qetfile
```

The snapshot is sorted alphabetically and includes a header comment:

```toml
# This file was generated by `qet snapshot`.

[[packages]]
qet_name = "@microsoft/vscode"
method   = "flatpak"

[[packages]]
qet_name = "@utils/htop"
method   = "apt"
```

> **Source:** [`qet/commands.py` — `get_snapshot_data()`](../qet/commands.py#L290-L303)

## Applying a Qetfile with `sync`

`sync` reads a Qetfile, compares it against the currently installed set, and installs any missing packages using the same logic as `qet add` (priority, fallback, security confirmation).

```bash
# Apply ./Qetfile in the current directory
qet sync

# Apply a Qetfile at a specific path
qet sync --file ~/dotfiles/Qetfile

# Skip all confirmation prompts (useful in bootstrap scripts / CI)
qet sync -y
```

**What `sync` does:**
1. Parses the Qetfile (defaults to `~/.config/qet/Qetfile`; override with `--file`).
2. Builds the current installed set by replaying the event log.
3. For each `[[packages]]` entry not yet installed, calls `add_package()`.
4. Prints a structured summary — installed ✓, skipped ─, failed ✗.
5. Exits with code `1` if any packages failed.

## Workflow: Bootstrapping a New Machine

1. On your **existing machine**, export your setup:

   ```bash
   qet snapshot > ~/dotfiles/Qetfile
   git -C ~/dotfiles add Qetfile && git commit -m "chore: update Qetfile"
   git push
   ```

2. On the **new machine**, install `qet`, then pull and apply:

   ```bash
   git clone https://github.com/you/dotfiles.git ~/dotfiles
   qet init
   # Copy or symlink the Qetfile, then:
   qet sync
   ```

## Workflow: Team-Shared Dev Environment

Create a `Qetfile` in your project repository to define all the CLI tools your team needs:

```toml
# tools/Qetfile

[[packages]]
qet_name = "@utils/ripgrep"
method   = "cargo"

[[packages]]
qet_name = "@web/nodejs"
method   = "apt"

[[packages]]
qet_name = "@python/uv"
method   = "script"
```

Any team member can install everything with:

```bash
cd tools/
qet sync
```
